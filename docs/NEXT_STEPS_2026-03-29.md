## 🟡 方向二：Search采集优化（Jina主力方案）

### 问题
- 当前Search成功率仅60%（HTML解析复杂、易过时）
- Bing页面结构经常变化，选择器需维护

### 解决方案
```
完全切换为Jina Reader作为主方案：

原方案：Bing HTML解析 (成功率60%) → Jina Fallback
新方案：Jina Reader主方案 (成功率95%) + 缓存 + 成本控制

特点：
  ✅ 自动处理JavaScript渲染
  ✅ 自动处理反爬
  ✅ 返回清晰的Markdown
  ✅ 支持智能截断（避免超长文本）
```

### 具体改动
- 修改 `collectors/search_collector.py`
- 移除HTML选择器逻辑（已优化但维护成本高）
- 强化Jina调用（当前是fallback，改为主方案）
- 实施搜索预算管理（P0企业优先）

### 预期效果
- Search成功率：60% → 85%
- 搜索成本：降低40%（Jina定价更优化）
- 维护成本：降低80%（无需维护选择器）

### 实施时间
- ⏱️ **5-7天**

### 推荐指数
⭐⭐⭐⭐ **很高**（釜底抽薪，解决Search采集核心问题）

## 🟢 方向三：本地LLM部署（成本与性能优化）

### 问题
- 完全依赖云LLM（DeepSeek/Qwen/Kimi）
- 成本：500-2000元/月（取决于采集量）
- 延迟：分析每条需要5分钟
- 对网络依赖强

### 解决方案
```
混合模型架构：

云LLM（Claude/Kimi）      本地LLM（Qwen-7B）
    ↓                           ↓
复杂决策价值判断        分类/评分/摘要提取
战略级分析                日常信息处理
(5%)                        (95%)

部署方案：
  1. 采购GPU服务器 (RTX4090 或 2×A100)
  2. 部署Qwen-7B-Chat (量化版4bit)
  3. 微调基础分类模型（可选）
  4. 集成到main.py的LLM路由层
```

### 成本分析
**初期投入**：硬件成本 2-5万元
**月度收益**：
- 云LLM成本：500-2000 → 100-200 (↓80%)
- ROI周期：3-6个月

### 实施时间
- ⏱️ **2-3周**
  - 硬件部署：3天
  - 模型配置：3天
  - 测试验证：7天

### 推荐指数
⭐⭐⭐⭐ **很高**（长期收益大，但需投资）

**何时开始**：建议在方向一、二完成后再做

## 🔵 方向四：实时告警系统（用户体验升级）

### 问题
- 当前周报形式，信息延迟3-7天
- P0企业重要信息（并购、专利、产品发布）无法实时推送
- 用户要等周五才能看到周报

### 解决方案
```
实时告警系统：

企业关键词 → 采集立即触发 → 实时分析 → Webhook推送
  (华为/中兴)        (≤10min)      企业微信/邮件

触发规则示例：
  ✅ P0企业 + importance≥4 + decision_value=true
  ✅ "并购"/"融资"/"新产品" 相关词
  ✅ 新增供应链/战略合作
```

### 实施方案
- 创建 `AlertManager` 类
- 在分析完成后立即check告警规则
- Webhook推送到企业微信
- 支持Web UI配置告警阈值

### 预期效果
- 重大信息响应时间：<1小时（vs当前3-7天）
- 用户关键信息不漏
- 提升产品价值感

### 实施时间
- ⏱️ **2周**

### 推荐指数
⭐⭐⭐ **中等**（锦上添花，但有助于用户粘性）
